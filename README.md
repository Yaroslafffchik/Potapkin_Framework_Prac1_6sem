# Библиотека треков - Практическая работа №1

Веб-служба для управления библиотекой музыкальных треков с конвейером обработки запросов.

## Описание проекта

Это ASP.NET Core приложение реализует мини веб-службу для хранения и управления каталогом музыкальных треков. Проект демонстрирует работу конвейера обработки HTTP-запросов с использованием middleware компонентов.

## Структура проекта

```
PotapkinPrac1/
├── Domain/              # Модели предметной области
│   └── Track.cs        # Модель музыкального трека
├── Services/           # Бизнес-логика
│   └── TrackService.cs # Сервис управления треками
├── Middleware/         # Компоненты конвейера обработки запросов
│   ├── RequestLoggingMiddleware.cs    # Логирование запросов/ответов
│   ├── ExceptionHandlingMiddleware.cs # Обработка исключений
│   └── TimingMiddleware.cs            # Измерение времени выполнения
├── Errors/             # Обработка ошибок
│   ├── ErrorResponse.cs        # Единый формат ошибок
│   ├── ValidationException.cs  # Ошибка валидации
│   └── NotFoundException.cs    # Ошибка "не найдено"
├── wwwroot/            # Статические файлы
│   └── index.html     # Веб-интерфейс
├── Properties/
│   └── launchSettings.json
├── Program.cs          # Точка входа и конфигурация
└── PotapkinPrac1.csproj
```

- .NET 10.0 SDK или выше
- Любой современный веб-браузер

## API Endpoints

### 1. Получить все треки
```
GET /api/tracks
```
Возвращает список всех треков

**Параметры запроса (опционально):**
- `artist` - фильтр по исполнителю (регистронезависимый поиск)

**Пример:**
```bash
curl http://localhost:8080/api/tracks
curl http://localhost:8080/api/tracks?artist=Queen
```

**Ответ (200 OK):**
```json
[
  {
    "id": 1,
    "title": "Bohemian Rhapsody",
    "artist": "Queen",
    "album": "A Night at the Opera",
    "duration": 354
  }
]
```

### 2. Получить трек по ID
```
GET /api/tracks/{id}
```
Возвращает трек с указанным идентификатором

**Пример:**
```bash
curl http://localhost:8080/api/tracks/1
```

**Ответ (200 OK):**
```json
{
  "id": 1,
  "title": "Bohemian Rhapsody",
  "artist": "Queen",
  "album": "A Night at the Opera",
  "duration": 354
}
```

**Ответ при ошибке (404 Not Found):**
```json
{
  "errorCode": "NOT_FOUND",
  "message": "Трек с ID 999 не найден",
  "requestId": "8c3f2a4d-5b1e-4f3a-9d2c-7e6f8a1b3c5d"
}
```

### 3. Создать новый трек
```
POST /api/tracks
Content-Type: application/json
```

**Тело запроса:**
```json
{
  "title": "Название трека",
  "artist": "Исполнитель",
  "album": "Альбом",
  "duration": 240
}
```

**Пример:**
```bash
curl -X POST http://localhost:8080/api/tracks \
  -H "Content-Type: application/json" \
  -d '{"title":"Test Song","artist":"Test Artist","album":"Test Album","duration":180}'
```

**Ответ (201 Created):**
```json
{
  "id": 4,
  "title": "Test Song",
  "artist": "Test Artist",
  "album": "Test Album",
  "duration": 180
}
```

**Ответ при ошибке валидации (400 Bad Request):**
```json
{
  "errorCode": "VALIDATION_ERROR",
  "message": "Название трека не может быть пустым",
  "requestId": "8c3f2a4d-5b1e-4f3a-9d2c-7e6f8a1b3c5d"
}
```

## Правила валидации

При создании трека проверяются следующие условия:
1. Название трека (`title`) не может быть пустым или содержать только пробелы
2. Длительность (`duration`) должна быть больше 0 секунд
3. Исполнитель (`artist`) не может быть пустым или содержать только пробелы

## Конвейер обработки запросов (Middleware Pipeline)

Каждый HTTP-запрос проходит через следующие этапы обработки:

### 1. TimingMiddleware
- Измеряет общее время выполнения запроса
- Записывает результат в журнал вместе с RequestId
- Позволяет отслеживать производительность

### 2. RequestLoggingMiddleware
- Генерирует уникальный идентификатор запроса (RequestId)
- Логирует входящий запрос (метод, путь, параметры)
- Логирует исходящий ответ (статус-код)
- RequestId доступен для всех последующих middleware

### 3. ExceptionHandlingMiddleware
- Перехватывает все исключения в конвейере
- Преобразует их в единый формат ErrorResponse
- Устанавливает правильные HTTP статус-коды:
  - 400 Bad Request - ValidationException
  - 404 Not Found - NotFoundException
  - 500 Internal Server Error - остальные исключения
- Включает RequestId для корреляции с логами

### 4. Endpoints (обработчики маршрутов)
- GET /api/tracks - получение всех треков (с опциональной фильтрацией)
- GET /api/tracks/{id} - получение трека по ID
- POST /api/tracks - создание нового трека

## Формат ошибок

Все ошибки возвращаются в едином формате:
```json
{
  "errorCode": "КОД_ОШИБКИ",
  "message": "Описание ошибки",
  "requestId": "уникальный-идентификатор-запроса"
}
```

Коды ошибок:
- `VALIDATION_ERROR` - ошибка валидации входных данных
- `NOT_FOUND` - запрашиваемый ресурс не найден
- `INTERNAL_SERVER_ERROR` - внутренняя ошибка сервера

## Логирование

Все запросы логируются в следующем формате:

```
Входящий запрос: RequestId=abc-123, Method=GET, Path=/api/tracks, QueryString=?artist=Queen
Время выполнения запроса: RequestId=abc-123, Duration=15ms
Исходящий ответ: RequestId=abc-123, StatusCode=200
```

При ошибках дополнительно логируется:
```
Ошибка обработки запроса: RequestId=abc-123, Message=Трек с ID 999 не найден
```

## Веб-интерфейс

Приложение включает в себя веб-интерфейс со следующими возможностями:
- Просмотр всех треков
- Фильтрация треков по исполнителю
- Поиск трека по ID
- Добавление нового трека
- Всплывающие модальные окна для отображения ошибок с полной информацией (код ошибки, сообщение, RequestId)

## Архитектурные решения

Подробности в ARCHITECTURE.md

### Разделение ответственности
- `Domain` - чистые модели предметной области без бизнес-логики
- `Services` - бизнес-логика и валидация данных
- `Middleware` - сквозная функциональность (логирование, обработка ошибок, метрики)
- `Errors` - типизированные исключения и формат ответов об ошибках

### Конвейер middleware
Middleware расположены в правильном порядке:
1. Timing - самый внешний, измеряет полное время
2. Logging - генерирует RequestId для последующих компонентов
3. ExceptionHandling - перехватывает ошибки из всех последующих этапов
4. Endpoints - бизнес-логика приложения

### Потокобезопасность
Используется `lock` для защиты от гонки данных при параллельных запросах к списку треков.

### Единый формат ошибок
Все ошибки возвращаются в одинаковом формате с RequestId для корреляции с логами.
