# Архитектурное описание проекта

## Обзор

Проект представляет собой веб-службу для управления библиотекой музыкальных треков, построенную на ASP.NET Core с использованием паттерна Minimal API. Основная архитектурная идея - демонстрация конвейера обработки HTTP-запросов через middleware компоненты.

## Архитектурные принципы

### 1. Разделение ответственности (Separation of Concerns)

Проект структурирован по слоям:

- **Domain** - Модели предметной области без бизнес-логики
- **Services** - Бизнес-логика и валидация данных
- **Middleware** - Сквозная функциональность (логирование, обработка ошибок, метрики)
- **Errors** - Типизированные исключения и формат ответов об ошибках

Каждый слой имеет четкую ответственность и не пересекается с другими.

### 2. Конвейер обработки запросов (Middleware Pipeline)

Порядок middleware критически важен для правильной работы системы:

```
HTTP Request
    ↓
[TimingMiddleware] ← Измеряет полное время выполнения
    ↓
[RequestLoggingMiddleware] ← Генерирует RequestId и логирует запрос/ответ
    ↓
[ExceptionHandlingMiddleware] ← Перехватывает исключения из всех последующих слоев
    ↓
[StaticFilesMiddleware] ← Обслуживает статические файлы (index.html)
    ↓
[Endpoints] ← Бизнес-логика приложения
    ↓
HTTP Response
```

**Обоснование порядка:**

1. **TimingMiddleware** - самый внешний, так как должен измерять полное время включая все остальные middleware
2. **RequestLoggingMiddleware** - генерирует RequestId на ранней стадии, чтобы он был доступен всем последующим компонентам
3. **ExceptionHandlingMiddleware** - должен перехватывать все ошибки, возникающие в endpoints и других middleware
4. **StaticFiles/Endpoints** - бизнес-логика находится в самом конце конвейера

### 3. Единый формат ошибок

Все ошибки возвращаются в едином формате `ErrorResponse`:

```json
{
  "errorCode": "КОД_ОШИБКИ",
  "message": "Описание ошибки",
  "requestId": "уникальный-идентификатор"
}
```

Это обеспечивает:
- Предсказуемость для клиентов API
- Возможность корреляции с логами через RequestId
- Типизацию ошибок через errorCode

### 4. Потокобезопасность

Используется `lock` в `TrackService` для защиты от гонки данных при параллельных запросах. Это необходимо, так как сервис зарегистрирован как Singleton.

## Компоненты системы

### TimingMiddleware

**Назначение:** Измерение времени выполнения запросов

**Принцип работы:**
1. Запускает Stopwatch перед вызовом следующего middleware
2. Ждет завершения обработки запроса
3. Останавливает Stopwatch и логирует результат с RequestId

**Почему это в middleware, а не в сервисе:**
- Измерение должно охватывать весь конвейер, а не только бизнес-логику
- Это сквозная функциональность, не связанная с предметной областью

### RequestLoggingMiddleware

**Назначение:** Логирование входящих запросов и исходящих ответов

**Принцип работы:**
1. Генерирует уникальный RequestId (GUID)
2. Сохраняет его в HttpContext.Items для доступа из других компонентов
3. Логирует информацию о запросе (метод, путь, параметры)
4. Вызывает следующий middleware
5. Логирует информацию об ответе (статус-код)

**Почему RequestId генерируется здесь:**
- Это единая точка входа для всех запросов
- RequestId должен быть доступен во всех последующих компонентах
- Позволяет коррелировать логи разных middleware для одного запроса

### ExceptionHandlingMiddleware

**Назначение:** Централизованная обработка исключений

**Принцип работы:**
1. Оборачивает вызов следующего middleware в try-catch
2. Перехватывает все исключения
3. Преобразует их в ErrorResponse с правильным HTTP статус-кодом
4. Логирует ошибку с RequestId
5. Возвращает JSON ответ клиенту

**Типы обрабатываемых исключений:**
- `ValidationException` → 400 Bad Request
- `NotFoundException` → 404 Not Found
- `Exception` (остальные) → 500 Internal Server Error

**Почему это в middleware:**
- Единая точка обработки ошибок для всего приложения
- Гарантирует единый формат ответов об ошибках
- Не требует дублирования логики try-catch в каждом endpoint

### TrackService

**Назначение:** Бизнес-логика управления треками

**Принцип работы:**
- Хранит список треков в памяти (List<Track>)
- Предоставляет методы CRUD операций
- Выполняет валидацию данных
- Использует lock для потокобезопасности

**Валидация:**
1. Название трека не пустое
2. Длительность больше 0
3. Исполнитель не пустой

**Почему валидация здесь, а не в middleware:**
- Валидация - это часть бизнес-логики предметной области
- Правила валидации могут быть сложными и зависеть от бизнес-контекста
- Middleware должны быть универсальными, а валидация специфична для каждой сущности

### Domain модели

**Track** - простая POCO модель без логики:
- Id: int
- Title: string
- Artist: string
- Album: string
- Duration: int (в секундах)

**Почему модели отделены:**
- Чистота модели предметной области
- Возможность использования в разных слоях
- Легкая сериализация/десериализация

### Errors (исключения)

**ValidationException** - ошибка валидации данных
**NotFoundException** - запрашиваемый ресурс не найден
**ErrorResponse** - DTO для ответов об ошибках

**Почему кастомные исключения:**
- Типизация ошибок
- Возможность различной обработки разных типов ошибок
- Семантическая ясность кода

## Принятые технические решения

### 1. Minimal API вместо контроллеров

**Обоснование:**
- Простота для небольших проектов
- Меньше boilerplate кода
- Явная настройка конвейера middleware
- Современный подход в ASP.NET Core

### 2. In-memory хранилище

**Обоснование:**
- Простота для учебного проекта
- Фокус на архитектуре, а не на персистентности
- Легкость тестирования
- Нет внешних зависимостей

**Ограничения:**
- Данные теряются при перезапуске
- Не подходит для production
- Ограниченная масштабируемость

### 3. Singleton для TrackService

**Обоснование:**
- Общее хранилище для всех запросов
- Простота управления состоянием
- Защита через lock

**Альтернативы:**
- Scoped - не подходит, так как каждый запрос получит свой экземпляр
- Transient - еще хуже, каждая инъекция создает новый экземпляр

### 4. GUID для RequestId

**Обоснование:**
- Гарантированная уникальность
- Не требует централизованного генератора
- Легко распознать в логах

## Безопасность

### Текущие меры:

1. **Валидация входных данных** - проверка на пустые строки и отрицательные числа
2. **Типизация ошибок** - клиент получает структурированные ответы, а не stack traces
3. **Логирование** - все запросы логируются для аудита

### Что нужно добавить для production:

1. **CORS** - ограничение доменов, которые могут вызывать API
2. **Rate Limiting** - защита от DoS атак
3. **Authentication/Authorization** - идентификация и авторизация пользователей
4. **Input sanitization** - защита от инъекций
5. **HTTPS** - шифрование трафика
6. **Validation на длину строк** - защита от переполнения памяти

## Масштабируемость

### Текущее решение подходит для:
- Учебных целей
- Прототипирования
- Локальной разработки
- До ~1000 треков в памяти

### Для масштабирования нужно:

1. **База данных** - переход на SQL/NoSQL вместо in-memory
2. **Кэширование** - Redis для часто запрашиваемых данных
3. **Асинхронность** - async/await для I/O операций
4. **Пагинация** - ограничение размера ответов
5. **Load Balancer** - распределение нагрузки между инстансами
6. **Message Queue** - для асинхронной обработки тяжелых операций

## Тестируемость

### Преимущества текущей архитектуры:

1. **Разделение на слои** - легко мокировать зависимости
2. **Dependency Injection** - явные зависимости
3. **Чистые модели** - легко создавать тестовые данные
4. **Типизированные исключения** - легко проверять обработку ошибок

### Что можно добавить:

1. Unit тесты для TrackService
2. Integration тесты для endpoints
3. Middleware тесты
4. Тесты валидации

## Выводы

Архитектура проекта построена на принципах:
- **Разделения ответственности** - каждый компонент делает одну вещь
- **Явности над неявностью** - конвейер middleware настроен явно
- **Расширяемости** - легко добавить новые middleware или endpoints
